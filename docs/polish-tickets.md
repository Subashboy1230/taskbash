## TICKET 1 — Add Sentry error tracking (client + server + Inngest)

**Priority:** P0 · **Effort:** Medium · **Status:** Planned

**Why it matters**

We are blind in production. There is no exception tracking of any kind — `@sentry/nextjs` is not installed, no config, no `withSentryConfig`. Langfuse covers LLM traces only; the `agent_events` table covers business events only. Unhandled exceptions in server actions, middleware, and Inngest crons (gmail-poll runs every 5 min; whatsapp/morning digests run hourly) currently surface only as console logs we never see. First sign of a prod bug today is manual QA or a user message.

**Scope (files to touch)**

- `package.json` — add `@sentry/nextjs`
- NEW `sentry.client.config.ts`, `sentry.server.config.ts`, `sentry.edge.config.ts`, `instrumentation.ts`
- `next.config.js` — wrap with `withSentryConfig`
- `inngest/functions/*.ts` — wrap `step.run` bodies in try/catch → `Sentry.captureException` (cover gmail-poll, morning-digest, whatsapp-morning-digest, whatsapp-meeting-reminder, whatsapp-meeting-scheduler)
- `middleware.ts:39-56` — try/catch around `supabase.auth.getUser()`, capture on failure
- The server-side auth/session load path — `Sentry.setUser({ id, email })` once per authenticated request
- `.env.example` — add `SENTRY_DSN`, `SENTRY_ORG`, `SENTRY_PROJECT`, `SENTRY_AUTH_TOKEN` placeholders
- NEW temp `app/sentry-test/page.tsx` (throws on render; delete after verifying)

**Acceptance criteria**

- [ ] `@sentry/nextjs` installed; three config files + `instrumentation.ts` present; `next.config.js` wrapped.
- [ ] Error-only tier: **session replay and `tracesSampleRate` performance monitoring disabled.**
- [ ] Every Inngest function captures exceptions to Sentry without swallowing the existing retry behavior.
- [ ] Authenticated requests tag the Sentry user (id + email).
- [ ] Hitting `/sentry-test` once produces an event in the Sentry dashboard within 60s; then the route is deleted.
- [ ] `SENTRY_*` vars documented in `.env.example` and the exact name/value list for Vercel prod env is printed.
- [ ] `tsc --noEmit && npm run build` clean. No em-dashes in any new UI string.

## TICKET 2 — Error boundaries across the app

**Priority:** P0 · **Effort:** Medium · **Status:** Planned

**Why it matters**

Only `app/today/error.tsx` exists. A throw anywhere in 9 other routes — or in the Sheet detail panel or calendar popover (the two most common throw sites) — blanks the whole page with React's default white screen. `react-error-boundary` is not a dependency, so nothing isolates a component-level throw from the layout.

**Scope (files to touch)**

- NEW `app/error.tsx` (Client Component, friendly "Something went wrong" + `reset()` "Try again", shadcn Card + dark-palette tokens)
- NEW `app/global-error.tsx` (renders its own `<html><body>`)
- NEW per-route `error.tsx` in: `app/handled`, `app/connections`, `app/activity`, `app/observability`, `app/network`, `app/settings/whatsapp`, `app/settings/functions`, `app/profile` (each: message + Try again + link back to `/today`)
- `package.json` — add `react-error-boundary`
- `app/today/today-shell.tsx:244-281` — wrap `DetailPanel` body in a client `<ErrorBoundary>`
- `app/today/today-calendar-column.tsx:410-430` — wrap calendar popover body in a client `<ErrorBoundary>`

**Acceptance criteria**

- [ ] Root `app/error.tsx` and `app/global-error.tsx` exist and render the dark-palette fallback.
- [ ] All 8 listed routes have an `error.tsx` with a Try-again button and a link to `/today`.
- [ ] A forced throw inside the detail panel or calendar popover degrades to a local fallback; the sidebar, layout, and calendar keep working.
- [ ] No raw Tailwind colors, no `text-[Npx]` literals, no em-dashes in any string.
- [ ] `tsc --noEmit && npm run build` clean.

## TICKET 3 — Fix snooze double-fire (form pending-guards)

**Priority:** P1 · **Effort:** Small · **Status:** Planned

**Why it matters**

Confirmed live bug: the Snooze picker trigger has no `disabled` state while the action is in flight, so a user can fire multiple snooze calls on one row. Same class of gap exists on a few other controls. (Note: the previously-suspected Add-task-clears and Send-Gmail-double-fire bugs were checked and are NOT present — already protected.)

**Scope (files to touch)**

- `app/today/today-view.tsx:1300-1350` — thread `isPending` into `SnoozeMenu`; disable `DropdownMenuTrigger` (line ~1329) and items while pending; guard the handler so a click during pending is a no-op
- `app/today/today-view.tsx:2219-2229` — Reject Draft button: add pending state + disable
- `app/today/today-view.tsx:~2062-2144` — edit-mode inline inputs (due date, description): disable while the commit is in flight
- `app/settings/whatsapp/whatsapp-settings-form.tsx:72-139, 207-208` — disable input fields and toggles while `pending`

**Acceptance criteria**

- [ ] Snooze trigger is disabled (or click is a no-op) while a snooze action is pending; rapid clicks produce exactly one server call.
- [ ] Reject Draft, edit-mode inline inputs, and WhatsApp form fields all disable during their in-flight action.
- [ ] No regression to Add-task or Send-Gmail-draft (both already guarded).
- [ ] No em-dashes in any error string.

## TICKET 4 — Custom 404 page + broken-image fallback

- NEW `components/ui/safe-image.tsx` (wraps `<img>`, `fallback` prop, swaps to neutral placeholder on `onError`)

- `app/_components/app-sidebar.tsx:84` — sidebar wordmark `/logo-new.png` → `SafeImage`
- `app/login/page.tsx:60` — Granola logo → `SafeImage`
- (Leave the user-screenshot blob `<img>` in `add-task-panel.tsx:483` as-is — local blob, low risk)

**Acceptance criteria**

- [ ] `/todayy` (any unknown route) renders the branded 404 with a working "Back to today" button, not the Next default.
- [ ] `SafeImage` exists and shows a placeholder (not a torn icon) when an image fails to load.
- [ ] All 4 brand logos, the sidebar wordmark, and the login logo render via `SafeImage`.
- [ ] No raw Tailwind colors, no em-dashes in any string.

## TICKET 5 — Surface silent load failures (loading/error/empty)

**Priority:** P2 · **Effort:** Small-Medium · **Status:** Planned

**Why it matters**

Skeletons and empty states are largely in place (added in today's commit). The remaining gap is that several page loaders use `.catch(() => [])`, so a failed fetch is indistinguishable from genuinely-empty data — the user (and we) cannot tell "nothing here" from "this broke." Calendar and observability fetches fail completely silently.

**Scope (files to touch)**

- `app/today/today-calendar-column.tsx:298-304` — day-event fetch: replace silent `.catch(() => setDayEvents([]))` with an error state + Retry
- `app/observability/page.tsx:22, 176-179` — surface loader failure in the Recent calls feed instead of rendering a blank table; add `app/observability/loading.tsx` skeleton for the per-prompt table
- `app/handled/page.tsx:20` and `app/activity/page.tsx:31-37` — distinguish load-error from empty (let the new route `error.tsx` from Ticket 2 catch real throws rather than swallowing)
- `app/profile/tabs/voice-tab.tsx:56`, `app/profile/tabs/prompts-tab.tsx:145` — show `Loader2` spinner on the regenerate / suggest actions

**Acceptance criteria**

- [ ] Calendar day-event load failure shows an error message + Retry, not a misleading "No events" state.
- [ ] Observability feed shows a clear "couldn't load calls" state on failure; the per-prompt table has a loading skeleton.
- [ ] `/handled` and `/activity` load errors are caught by a route error boundary (depends on Ticket 2), not swallowed to `[]`.
- [ ] The `/activity` Suspense boundary has a skeleton fallback.
- [ ] Voice-regenerate and prompt-suggest buttons show a spinner while pending.
- [ ] No em-dashes in any new UI string.
