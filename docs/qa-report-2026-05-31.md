## P0 — Data correctness or visible-content breakage

### P0-1. React hydration mismatch on /today (Minified React error #418)
Every page load of /today throws a client-side exception:

> Error: Minified React error #418; visit https://react.dev/errors/418?args[]=text — text content did not match.

#418 is the "server rendered text didn't match client" hydration error. This means at least one text node is computed differently on the server than on the client (typical causes: `new Date()` rendered without a stable formatter, `Math.random`, locale differences, or a client-only state value reflected during SSR). The error is thrown on every load — confirmed at 1:44 PM and again at 1:49 PM. The whole subtree subsequently re-renders, which kills SSR's perceived speed and may flash content. Reproduce: load taskbash.app/today, open DevTools console.

### P0-2. Subtask duplication on the SpendHound task (10 subtasks, ~4 distinct)
On the P0 task **"Discuss SpendHound platform with CEO and request QuickBooks admin access"** the panel shows 10 subtasks, but inspection reveals near-duplicates:

- "Contact CEO upon return from India (Saturday) to schedule discussion about SpendHound benefits"
- "Schedule meeting with CEO upon return from India (Saturday) to discuss SpendHound benefits" (same intent, different phrasing)
- "Schedule meeting with CEO upon return from India (Saturday) to discuss SpendHound benefits" (duplicate row)
- "Request QuickBooks admin access from CEO for platform integration setup"
- "Request QuickBooks admin access from CEO for platform integration"
- "Request QuickBooks admin access and Brex admin access from CEO for platform integration"
- "Confirm Thursday 2:30 PM follow-up call with SpendHound specialist for 5-10 minute dashboard integration"
- "Confirm Thursday 2:30 PM follow-up call with SpendHound specialist for integration"
- "Confirm Thursday 2:30 PM follow-up call with SpendHound specialist for 5-10 minute setup"

Three of these are near-identical "schedule meeting with CEO" rewordings; three more are "Confirm Thursday 2:30 PM call" rewordings. Subtask dedup is not running (or its semantic-hash threshold is too loose). This is the marquee feature of taskbash, and the marquee task on the list is showing the bug front-and-center.

### P0-3. Apostrophes rendered as `&#39;` HTML entity literals
On /today, subtitle text shows literal `&#39;` instead of `'`. JS audit confirms 3 occurrences right now:

- "25/2026 season of Technovation Girls. We&#39;re at 90% of..."
- "Hi Jacob, Thanks for checking in. I&#39;m going back through these and will pull..."
- "update on the active conversations. I&#39;ll circle back shortly..."

This is double-encoding: Gmail's HTML body is HTML-decoded into our store, but something downstream encodes the result again (or stores already-encoded entities and renders them via a plain text node). The same string also appears on /today in the Unread tab ("I&#39;d love to get you...", "I&#39;m checking in", "it&#39;s important"), so the bug is in the shared subtitle/preview pipeline, not in one component.

### P0-4. Open count mismatch between /today (74) and /profile (82)
The /today "Open" tab pill shows 74. The /profile Overview tile says "82 Open tasks." Difference = 8, which matches the Prep tab count exactly — meaning /profile is summing Open + Prep and labelling the total "Open tasks", while /today treats Prep as its own bucket. Pick one definition and use it on both surfaces (recommend: Open + Prep = "open" on /profile, OR label /profile tile "Open + Prep = 82").

### P0-5. Voice tab content contains em-dashes despite the em-dash ban
The /profile → Voice tab renders learned Writing Voice prose containing multiple em-dashes, e.g.:

- "...without being stiff."
- "...used universally across cold outreach, warm relationships, and internal staff alike."
- "Shorter replies use no formatting — plain prose only, typically one to two short paragraphs."
- "Standard opening: 'Hi [First name],' — used universally..."

The em-dash ban (task #115) is supposed to apply to UI strings and AI prompts. Either the Voice generator prompt is missing the "no em-dashes" instruction, or the post-process strip step isn't running on this surface. This is also unintentional irony — the user-visible style guide itself violates the style rule.

## P1 — Visible UI defects, content quality

### P1-1. Slop-rate chart on /profile → Stats is empty
The "Slop rate by source (last 30 days)" chart renders the axes, gridlines (0%, 25%, 50%, 75%, 100%), date labels (05-13 → 05-31), and the source legend (calendar, gmail, granola, linear, manual) — but no lines or points are plotted. Either there's been no slop feedback (in which case the chart should display an empty-state message, not look broken), or the loader is returning data the chart can't consume. Compare against the /profile Stats card claim of "846 Cleared this month" — there's enough activity to expect some series data.

### P1-2. Subtitle for Gmail items is raw email preview text + boilerplate
The "Reply to Technovation Judges re: 🎃 A few more hours to Judge" row's subtitle reads:

> "help us reach 100% View this email in your browser You are receiving this email because you are registered as a Technovation Judge for the 2025/2026 seas..."

This is the email's preheader + boilerplate ("View this email in your browser You are receiving this email because..."), not a clean summary. Subtitle should be a concise LLM summary, or at minimum strip "View this email in your browser" / "You are receiving this email" boilerplate.

### ~~P1-3. /profile → Prompts shows "no data" for all 7 prompts~~ — RETRACTED
Tester missed that each row is an expandable accordion (chevron expands to show real prompt body + stats). The "no data" label is the collapsed-row summary, not an empty state. Working as intended.

### P1-4. "Generating description and subtasks…" lingers briefly even when description and subtasks already exist
Opening the Technovation Reply panel showed a "Generating description and subtasks..." spinner row even though the description and subtasks then materialized within ~2 seconds. The state is real (subtasks went 0 → 3 and description populated mid-screenshot), but the spinner placement is awkward: it sits below the draft + Send button and lingers next to a fully-rendered Subtasks block. Loader should disappear the instant any subtask renders, not co-exist with them.

### P1-5. Open count jumps mid-session (71 → 74) with no visible action
While the detail panel was open and only a tab in the panel was clicked, the Open pill jumped from 71 to 74. The most likely cause is that 3 newly-generated subtasks for the Technovation reply were inserted as top-level items in `items` and therefore counted as "Open" — see also P1-7. Counts should not change while the user is reading a single task.

### P1-6. /activity → Tasks tab is noisy with subtask creation events
The Tasks feed at 1:46 PM shows three rows:

- Found "Submit scores before end of business today"
- Found "Complete any partially finished scores from previous sessions"
- Found "Log into my.technovationchallenge.org and start a new project score"

These are the three subtasks of one parent task (the Technovation reply). Each got its own `task_events.created` row. The Activity feed should either suppress subtask creations, group them under their parent ("Found 3 subtasks for 'Reply to Technovation Judges...'"), or render them indented under the parent event.

### P1-7. Subtasks appear to be inserted as top-level items (drives P0-4 and P1-5)
Tying the threads of P1-5 + P1-6: subtasks of a parent task seem to be persisted as standalone `items` rows that count toward `Open`. That explains the jump and the activity-feed noise. If that's intentional, the count should exclude subtasks; if not, subtasks should live on the parent and not get their own `items` row.

### P1-8. /today greeting "Heavy day. Triage the top of the list first." stays even when viewing Cleared or Unread tabs
The same hero heading shows on every tab. On the Cleared tab — which is 3 items, mostly checked off — "Heavy day. Triage..." is wrong. The greeting should react to the active tab (Cleared → "Nice work — N items cleared today"; Unread → "13 unread threads need a look", etc.) or only show on the Open tab.

Same chip styling is reused in the detail panel's "FUNCTIONS" section — that makes it unclear which functions are *assigned to this task* vs. which are simply *available*. Pick a selected style (filled background / ring / checkmark) that's distinct from the always-on color.

### P2-5. "Unread" tab rows redundantly show a green "Unread" pill
Every row in the Unread tab carries a status pill labelled "Unread." Tab is named Unread; pill is redundant. Drop the pill or swap it for sender / time-since-arrival.

# Round 2 — Action-path findings

Tested live by clicking through every action: Add task (with submit), pencil edit, subtask check/add, function chip toggle, snooze, slop, dismiss, mark as done, open-in-gmail, re-run tasks.

## P0 — Data correctness or breakage (Round 2)

### P0-6. Subtask-as-item bug is a *cascading explosion*, not a one-off
P1-7 in the first report flagged that subtasks get persisted as top-level `items`. Round 2 confirms it's worse: those leaked items then themselves generate descriptions + subtasks on first open, and **those grandchild subtasks also get persisted as top-level items**. Live trace:

1. I added one manual task: "QA test task — please ignore" → Open went 75 → 77 (parent + 2 LLM-generated subtasks).
2. The 2 leaked subtask-items were now visible in the list ("Verify JSON output format..." and "Confirm task generation system...").
3. I opened "Submit scores before end of business today" (itself a leaked subtask from the morning Technovation reply extraction). On open, it generated 4 new subtasks AND 4 new top-level items — "Confirm receipt and send completion notification", "Enter or upload scores into the submission system", "Verify scores for accuracy and completeness", "Gather all score sheets...".
4. I opened "Verify JSON output format..." (also a leaked subtask). On open, it generated **another 4** subtasks + 4 top-level items: "Check for em-dashes and replace with hyphens or colons", "Verify subtasks array has 2-4 items, each starting with a verb", "Confirm description field contains 1-2 sentences with specific context", "Validate JSON syntax using a linter or parser tool".

So each opened leaked-subtask-item creates ~4 more leaked items. Open count went 75 → 77 → 80 in three opens. Each open round-trips to Claude to generate the subtasks. This is unbounded growth driven entirely by clicking around. The fix:

- The on-open "Generating description and subtasks…" job should not write subtasks to `items`. Subtasks belong on the parent (whatever column holds them).
- Audit existing `items` for orphaned source='manual' rows whose semantic_hash matches the subtask text of another item, and reparent or delete them.
- Add a guard: don't auto-generate subtasks for an item that already has a parent_item_id, or whose source is "subtask".

### P0-7. Pencil-edit title save updates the row but not the open panel header
Reproduce: open detail panel → click pencil → change title → Save. Result:

- The row in the list updates to the new title (good).
- The panel header (top right) keeps the *old* title until the panel is closed and reopened.

Two render paths for the same field; they need to share state. Side effect: the title looks like it didn't save, until the user closes the panel.

### P0-8. Manual task on submit creates not just the task but also LLM-generated subtasks as top-level items, with absurd content
I typed title "QA test task — please ignore" with no description. The system:

- Inserted the parent task (good).
- Synchronously generated a description: *"This is a QA test task to verify the task generation system is working correctly. No actual work is required — this task should be ignored and closed."*
- Generated 2 subtasks: *"Confirm task generation system processed input correctly"* and *"Verify JSON output format is valid and properly structured"*.

Two failures here:

1. The description says "this task should be ignored and closed" while the subtasks say "verify the system works." The LLM is fighting itself — title told it to ignore, but it ran the subtask generator anyway.
2. Both subtasks then got persisted as top-level items (per P0-6) → list shows 3 new items instead of 1.

The subtask generator prompt is clearly overfitting on QA / meta-task language. It read "test task" as "verify the test infrastructure" instead of "this is a test, ignore it."

### P0-9. Function-chip click in detail panel crashes the page with `TypeError: network error`
Reproduce: open any detail panel → click an unselected function chip (I clicked "Product"). Result:

- Full-page error boundary: *"Something broke rendering /today. network error. TypeError: network error."*
- "Try again" button does not recover — same error reappears immediately.
- Only a hard navigation to `/today` restores the page.

A Server Action that should add a function tag is throwing a client-side network error, and the error boundary catches it at the *page* level instead of the *panel* level — so a single failed chip click nukes the whole view. Two bugs in one:
- (a) the chip-click action is failing (likely the assignment endpoint).
- (b) the error boundary scope is too wide; a failed function tag should not destroy /today.

### P0-10. "Open in Gmail" doesn't open the saved draft — it opens a fresh `mail.google.com/?fs=1&to=...&body=...` compose URL

**Update 2026-05-31:** The Send-now half of this bug is fixed. `executeProposedAction` was reworked to use `lib/gmail/drafts.ts → createGmailDraft + sendGmailDraft`, with `In-Reply-To` and `References` headers in the MIME, so threading is preserved. Manual prereq before it'll work in prod: add `gmail.modify` scope in the Nango dashboard and re-auth Gmail. The Open-in-Gmail half (the actual button that opens a tab) is still URL-mailto compose; see fix plan §E.6.

The auto-drafts PRD (the one I just reverted) is the long-term fix; right now, *Open in Gmail* opens this URL:

```
https://mail.google.com/mail/u/0/?fs=1
  &to=volunteers@technovation.org
  &su=Re:+%E2%8F%B0+A+few+more+hours+to+Judge
  &body=Hi+Megan%2C%0A%0AI+appreciate+the+reminder...
  &tf=cm
```

That's a URL-mailto compose, not a draft. Two consequences:

- (a) The reply is **not threaded** — Gmail builds a brand-new message, ignoring the original thread, because there's no `In-Reply-To` or `References` header carrier. Hitting Send in Gmail will land in the recipient's inbox as a fresh email with subject "Re: ⏰ A few more hours to Judge", *not* in the original thread.
- (b) There's no Gmail draft saved. If the user navigates away from the Gmail compose tab without sending, the body is lost; nothing for the user to come back to in Gmail Drafts.

The button label "Open in Gmail" promises a saved draft. It delivers a one-shot compose URL.

## P2 — Inconsistency / polish (Round 2)

### P2-6. Slop menu wording is good but Slop ↔ Trash icon coding is misleading
The trash-can icon on hover *looks* like a hard delete. The menu it opens is actually "Why is this slop?" — a feedback collector. Two problems:

- (a) Users will hesitate to click the trash because it looks destructive.
- (b) Once the menu is open, the labels are good ("Repeat — Already exists as another task" reads well), but there's no preview of what happens on the row (it's marked completed + a feedback row is written to `item_feedback`). Add a one-line tooltip on the trash icon: "Why is this slop?" or "Mark as slop."

### P2-7. Function chips in detail panel: selected-state contrast is only ~20% darker fill
The first report (P2-4) said "no selected state." On closer look there *is* a state — selected chips fill in (darker bg, black text), unselected stay outlined. But the difference is subtle enough that I and the user above both missed it on first viewing. Suggest a checkmark ✓ or "1.5x ring" on selected chips, in addition to the bg fill.

### P2-8. P2-3 source naming inconsistency repeats: list row says "From manual" for items that source=manual, but Add task and the Connections page never use the word "manual"
"Manual" is the internal source name. Users see it on row subtitles ("From manual") but never see it as a category anywhere they could action it. Either drop it from the row ("Added by you" reads better) or surface it elsewhere consistently.

## Live bug exploit — repro recipe for the cascading-subtask explosion

If you want to watch the cascade on prod live, here it is end-to-end (takes ~30 seconds, runs ~6 Claude calls):

1. Go to `/today`. Note the Open count.
2. Click "Add task." Title: `Test the system`. Submit.
3. Open count +3 (1 parent + 2 leaked subtasks).
4. Click into one of the leaked subtask-items (e.g. "Confirm description field contains correct information").
5. Wait 2 seconds for "Generating description and subtasks…" to resolve.
6. Open count +4 more (4 grandchildren leaked into items).
7. Click into one of *those* grandchildren. Open count +4 more (great-grandchildren).
8. Stop when the user gets tired or hits the Anthropic rate limit.

Cleanup script suggestion: `delete from items where source = 'manual' and (parent_item_id is not null or extraction_meta->>'auto_generated_subtask' = 'true')`.

## P0 — Data correctness or breakage (Round 3)

### P0-11. Empty-title Add task submit fails silently
Open Add task → leave Title empty → click Create task. The submit doesn't go through (good), but there's also no inline error message, no toast, no field-level red text. The user clicks Create and nothing happens. They have no idea why.

Native HTML5 `required` attribute appears to be blocking, but the form swallows the browser's default constraint-validation popup. Add a visible inline error: "Title is required" under the field on first failed submit.

### P0-12. Subtask "Mark all done" does not auto-complete the parent
With the QA test task EDITED, I deleted one subtask leaving "1 of 1 done." The remaining subtask was already checked. The parent task stayed `status='open'` and did not auto-complete. If the pattern is "subtasks finish → parent done," it isn't implemented. If the pattern is "parent stays open until the user explicitly closes it," that's fine but you should remove the "1 of 1 done" pill from looking like a completion indicator (it currently *looks* like the task is done from the row).

Either:
- (a) When all subtasks are checked, auto-set parent to `status='completed'` and write a `task_event` of kind 'auto_completed_via_subtasks'.
- (b) Or pop a toast: "All subtasks done. Mark task as Done?"

### P0-13. `draft.followup` prompt has run 365 times today but nothing surfaces in /today
/observability shows `draft.followup v1` is the highest-volume prompt today: 365 calls, $0.79, 3048ms avg, 0 errors. It's twice the volume of `extract.gmail` (242 + 67 v2 = 309). But the /today UI has no "Follow-up" tab, no follow-up status on rows, no follow-up surface anywhere I saw. Either:
- (a) Follow-ups are being generated but discarded (waste of $0.79/day and 695k tokens).
- (b) Follow-ups are stored but never read by any UI loader.
- (c) Follow-ups *do* surface and I missed them.

Worth checking which it is. At 365 calls/day, this is the most expensive prompt in the system.
