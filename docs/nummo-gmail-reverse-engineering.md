# Nummo Gmail surfacing: reverse-engineering + taskbash implementation spec

Hand this to Claude in Cursor as the source-of-truth for shipping the "Gmail feels real-time + every task is high-signal" experience that Nummo gets right and taskbash currently doesn't.

The user has 184 open items right now, most of them stale. The goal is not just to *match* Nummo's polish but to make every new Gmail message either become a task within a minute of arrival or get silently dropped, with no human triage in between.

---

## 1. What Nummo does right (from screenshots)

### Visual pattern

Each task row is two lines:
- **Bold imperative title** describing the action, not the source.
  - `Reply on EverTutor pilot next steps`
  - `Reply with call options`
  - `Reply on Edlink next steps`
  - `Review Kemi's profile and test the API flow`
  - `Reply to EdSails on status`
- **Synthesized subtitle** in muted gray, 1-2 sentences, explaining WHO triggered it, WHAT they're asking, and WHY it matters now.
  - `Shivam shared the updated EverTutor-specific GTM report and outlined pilot requirements: ICP, message cop...`
  - `Klevi reactivated the key and this is an ongoing beta-feedback thread. Draft a reply suggesting a 30-minute c...`
  - `Danny sent a technical recap and proposed next steps, including getting SigIQ set up with Edlink sandbox acc...`
- **Right-side status pill**:
  - `Drafted a reply` (green, default state for tasks with a pre-draft)
  - `Promised in your reply` (different tone, surfaces commitments YOU made in an outbound email)
- **Top-of-list pleasantry**: `A few left. Let's clear them.` — short, contextual greeting + count badge. Not a generic `Good afternoon, Subash`.

### Completed-today section

- Collapsible header: `COMPLETED TODAY 4 View All`
- Same two-line row format for completed items
- Status pills track lifecycle:
  - **Completed** (blue) — generically marked done
  - **Approved** (green) — user sent the drafted reply
  - **Rejected** (red) — user dismissed the draft
- Inline @-mention chips: `@Hollins P`, `@Mentormatch`, `@Re: Setting up demo for Mentor Match` — entities (people, projects, threads) get pill chips inline in the subtitle so the user can scan relationships fast.

### Behavioral patterns inferred from the data

1. **Real-time-ish Gmail.** Items surface as new emails arrive, not just on a daily cron. Probably 1-5 minute polling or Gmail push notifications via Pub/Sub.
2. **Commitments-from-sent-folder.** `Promised in your reply` means they extract from the user's SENT messages too. When Subash writes "I'll send the doc Tuesday", that becomes a task.
3. **Thread context aware.** Subtitles reference what happened in the thread ("Klevi reactivated the key and this is an ongoing beta-feedback thread"). They don't just summarize the latest message — they summarize the trajectory.
4. **Entity extraction.** People (Hollins P, Karttikeya, Klevi), projects (EverTutor, EdSails, Mentor Match, CALSA), and thread references all get extracted as first-class entities and rendered as chips.
5. **Specific over generic.** Titles include the *thing*: "Reply with call options" not "Reply to Klevi". "Reply on EverTutor pilot next steps" not "Reply on email".
6. **Reply lifecycle tracking.** Drafted → Approved/Rejected/Completed. Three terminal states, not two. Rejected is important because it captures "I read it and chose not to reply" as data.

### What Nummo skips (they're not great at this)

- Granola integration is weak per the user. Probably means meeting commitments don't make it onto the list as crisply as email replies.
- Function/category tags are absent in the screenshot — no Product/Ops/QA chips. They lean on the synthesized subtitle to convey context, not on tags.
- No priority pills (no P0/P1/P2). Implicit ordering.

## 2. What taskbash currently does (gap analysis)

### Done correctly
- ✅ Per-thread Claude extraction (`lib/extract/gmail.ts`)
- ✅ Per-thread reply drafting (`lib/draft/reply.ts`)
- ✅ `proposed_action` field stores the drafted reply
- ✅ Source ref dedup (gmail_thread_id + gmail_message_id)
- ✅ Cleared-task suppression
- ✅ Function tags via classifier
- ✅ Slop / approve / dismiss lifecycle

### Missing or weak

1. **Polling is on-demand only.** `requestRefresh` runs when the user clicks "Re-run tasks". The Inngest cron runs daily. There's NO "your inbox got a new email 30 seconds ago, here's a task" path. Everything is batched.

2. **Title quality.** The Gmail extractor prompt asks for an imperative title but doesn't require the thing-being-acted-on. We get "Reply to Megan" instead of "Reply to Megan about Ramp interest". Nummo's titles always include the topic.

3. **No subtitle.** The current row shows `Auto-extracted from Gmail — <thread subject>`. That's source metadata, not synthesized context. We need a 1-2 sentence "why this matters now" line generated by Claude at extract time, persisted to a new field, rendered in the row.

4. **No commitment extraction from SENT folder.** The Gmail extractor only reads `in:inbox`. It misses commitments the user made in outbound replies.

5. **No reply lifecycle states.** We have `status` (open/completed/dismissed/snoozed) and `proposed_action` (the draft). We don't track "the user reviewed the draft and rejected it" vs "sent it" vs "did neither". Nummo distinguishes Approved / Rejected / Completed.

6. **No entity extraction.** People mentioned in threads aren't extracted as entities. We'd need a `entities` table (or jsonb) + UI chips.

7. **The greeting is generic.** "Good afternoon, Subash" — not contextual to the actual list state. Nummo uses count-aware copy like "A few left. Let's clear them." or "All clear" or "Heavy morning today".

## 3. Implementation plan

Six work blocks, ordered by impact-to-effort ratio. Each block lists files to touch, DB changes, prompt changes, UI changes. Ship one block end-to-end before starting the next.

---

### Block A — Real-time Gmail polling (highest impact)

**Goal:** new email arrives → task appears within 60 seconds.

**Approach options:**

**Option 1: Frequent polling (simplest, ship first).**
Hit Gmail every 60-90 seconds via an Inngest scheduled function. Use `historyId` to fetch only what changed since the last poll instead of re-running the full extractor each time.

- New file: `inngest/functions/gmail-poll.ts`
- New table: `gmail_sync_state` with `user_id`, `last_history_id`, `last_polled_at`
- New extractor entry point: `extractGmailActionItemsIncremental({ userId, sinceHistoryId })` that uses Gmail's `users.history.list` endpoint
- Cron: `* * * * *` (every minute) with rate limiting per user

Pros: easy to ship, no Gmail Pub/Sub setup, works locally.
Cons: still polling (latency ~30-60s avg), Inngest billing per invocation.

**Option 2: Gmail push notifications (proper real-time).**
Gmail watch + Google Cloud Pub/Sub. Webhook fires within seconds of email arrival.

- Setup: enable Pub/Sub topic, configure Gmail watch via `users.watch`, expose a webhook at `/api/gmail-push` that reads the notification, fetches the changed message, runs extractor
- Watch requests expire after 7 days, need to renew on a daily cron
- Requires GCP project + Pub/Sub topic + service account

Pros: actual real-time, no polling cost.
Cons: more setup, GCP IAM headache, harder to dev locally.

**Recommendation:** Ship Option 1 first (week-1 work), migrate to Option 2 later if polling cost becomes real.

**Files to add/change:**

```
inngest/functions/gmail-poll.ts          NEW
lib/extract/gmail-incremental.ts         NEW (uses users.history.list)
migrations/014_gmail_sync_state.sql      NEW
app/api/inngest/route.ts                 register new function
```

**Migration sketch:**

```sql
create table gmail_sync_state (
  user_id uuid primary key references auth.users(id) on delete cascade,
  last_history_id text not null,
  last_polled_at timestamptz not null default now(),
  watch_expires_at timestamptz,  -- for push approach later
  updated_at timestamptz not null default now()
);
```

### Block B — Better titles + synthesized subtitles

**Goal:** every task row reads like Nummo's: specific imperative title + one-line synthesized why.

**Prompt changes — `lib/extract/gmail.ts`:**

Update the system prompt to require:
- Title: imperative + the SPECIFIC topic, max 8 words. Examples: `Reply on EverTutor pilot next steps`, `Send Linear roadmap doc to Kartik`. NOT `Reply to email` or `Reply to Megan`.
- A NEW field: `subtitle` (string, 1-2 sentences, max 25 words) explaining who triggered it, what they're asking, and what context the user needs. Examples from Nummo:
  - `Shivam shared the updated EverTutor-specific GTM report and outlined pilot requirements: ICP, message copy, target school list.`
  - `Klevi reactivated the key and this is an ongoing beta-feedback thread. Draft a reply suggesting a 30-minute call.`

**Schema change — `migrations/015_item_subtitle.sql`:**

```sql
alter table items add column subtitle text;
```

**Type change — `lib/types.ts`:**

Add `subtitle?: string | null` to `ExtractedItem` and `Item`.

**Persist change — `lib/digest/run.ts`:**

In the insert block, add `subtitle: fresh.subtitle ?? null`.

**UI change — `app/today/today-view.tsx`:**

Find the task row component. Currently the metadata line under the title reads:
```tsx
Auto-extracted from Gmail — {item.parent_context}
```
Replace with:
```tsx
{item.subtitle ?? `From ${SOURCE_LABEL[item.source]}: ${item.parent_context}`}
```

**Backfill:** existing rows have no subtitle. Either:
- Run `scripts/backfill-subtitles.ts` (call Claude on each existing item's source_excerpt to generate one)
- Or just let the next extraction sweep fill in new items and existing items keep the fallback "From Gmail: ..."

### Block C — Commitments from SENT folder

**Goal:** when Subash writes "I'll send the deck Tuesday" in a reply, that becomes a task.

**Extractor change — `lib/extract/gmail.ts`:**

Currently `query = 'in:inbox newer_than:${days}d -category:promotions -category:social'`. Add a second query for SENT:
```ts
const sentQuery = `in:sent newer_than:${days}d`
const sentThreads = await listThreads(sentQuery)
for (const ref of sentThreads) {
  const thread = await fetchThread(ref.id)
  const items = await extractItemsFromThread(thread, userEmail, { mode: 'commitment' })
  items.push(...items)
}
```

**Prompt fork.** `extractItemsFromThread` already takes a thread. Add a `mode` parameter that switches the system prompt:
- `mode: 'inbox'` — current behavior. Extract action items the user OWES based on incoming messages.
- `mode: 'commitment'` — NEW. Look at the user's outbound message and extract any explicit promises they made: "I'll send X by Y", "I'll loop in Z", "I'll get back to you on Q". Each promise becomes a task with `tag: 'commit'`.

**Status pill — UI change:**

Add a new `status_label` value: `'Promised in your reply'` (Nummo's exact copy). Style: amber, indicating self-imposed commitment. Renders when `item.tag === 'commit' && item.source === 'gmail' && item.source_ref.sent_by_user === true`.

**Source ref addition:**

```ts
source_ref: {
  gmail_thread_id,
  gmail_message_id,
  sent_by_user: true,  // NEW, indicates commitment from sent folder
}
```

### Block D — Reply lifecycle: Approved / Rejected / Completed

**Goal:** track the three terminal states for tasks with a drafted reply.

**Schema change — `migrations/016_reply_lifecycle.sql`:**

```sql
alter table items add column reply_outcome text;
-- one of: 'approved' | 'rejected' | 'completed' | null
-- 'approved' = user clicked Send on the draft
-- 'rejected' = user clicked Dismiss on the draft without sending
-- 'completed' = user marked done some other way (e.g. replied outside taskbash)
```

**Server action changes — `app/today/actions.ts`:**

- `sendDraft(itemId)` — already exists for Gmail send. Add `reply_outcome: 'approved'` on success.
- `dismissDraft(itemId)` — NEW. Sets `status='completed'`, `completed_at=now()`, `reply_outcome='rejected'`. Renders as "Rejected" pill in /handled.
- `completeItem(itemId)` — already exists. If item has `proposed_action` and `reply_outcome` is null, set `reply_outcome='completed'` (the generic "I marked it done").

**UI change — DetailPanel / row hover-triage:**

Today the row has a check (Complete) + X (Dismiss). Add a third subtle action when `proposed_action` exists: "Reject draft" → calls `dismissDraft`.

**Completed-today rendering — `app/today/today-view.tsx` ClearedTab:**

Render the pill based on `reply_outcome`:
- `approved` → green "Approved" pill
- `rejected` → red "Rejected" pill
- `completed` or null → blue "Completed" pill (current default)

### Block E — Entity chips in subtitle

**Goal:** people + projects + thread refs render as inline pill chips like Nummo's `@Hollins P`, `@Mentormatch`.

**Approach:**

At extract time, ask Claude to return an `entities` array alongside the task:
```json
{
  "title": "...",
  "subtitle": "Karttikeya forwarded the CALSA Summer Institute invite and wrote 'Let's go here.'",
  "entities": [
    { "kind": "person", "label": "Karttikeya", "ref": "karttikeya@..." },
    { "kind": "project", "label": "CALSA Summer Institute" }
  ]
}
```

**Schema change — `migrations/017_item_entities.sql`:**

```sql
alter table items add column entities jsonb;
-- jsonb array of { kind, label, ref? }
```

**Rendering:**

In the task row's subtitle, run a small post-processor that scans for each entity's `label` substring and wraps it in a `<EntityChip>` component. Style: rounded pill, faint border, dot color based on `kind` (person=green dot, project=blue dot, thread=orange dot — matches Nummo).

This is the polish that makes the list scan-able. Defer until A-D are working.

### Block F — Contextual greeting

**Goal:** replace `Good afternoon, Subash` with state-aware copy.

**Logic — `lib/load-digest.ts`:**

After loading open + completed counts, pick a greeting from:

```ts
function pickGreeting(open: number, completed: number, hour: number) {
  if (open === 0) return "All clear. Beautiful."
  if (completed > 0 && open < 5) return "A few left. Let's clear them."
  if (open > 30) return "Heavy day. Triage the top of the list first."
  if (hour < 12) return "Morning, Subash. Here's what's queued."
  if (hour < 18) return "Afternoon check-in."
  return "Late, but still time to clear a few."
}
```

Render as `digest.greeting`. Already plumbed through `MockDigestSummary`.

1. **Block B (titles + subtitles)** — ship this first. It's the single biggest visual quality jump and doesn't need any new infra.
2. **Block F (greeting)** — 10 minutes of work, frees up the top-of-page from a generic copy.
3. **Block A (real-time polling)** — biggest behavioral change. Needs new Inngest function + sync state table.
4. **Block C (sent-folder commitments)** — high signal but moderate effort. Needs a new prompt + extractor mode.
5. **Block D (reply lifecycle)** — important for the "Approved / Rejected" UX but it's a quality-of-life refinement, not core functionality.
6. **Block E (entity chips)** — polish. Ship last.

Each block can be reviewed + merged independently. Don't try to ship all of them in one PR.

## 9. Risks + mitigations

| Risk | Mitigation |
|---|---|
| Real-time polling burns Anthropic credits (Claude call per thread per minute) | Throttle to once per unique thread per hour; only call Claude when `historyId` shows a new message in that thread |
| Subtitle inflation: Claude writes 50-word subtitles | Hard constraint in the prompt + a server-side truncate to 200 chars |
| Sent-folder commitments duplicate inbox-extracted commitments | Dedup by gmail_thread_id; if both an inbox extraction and a sent extraction land for the same thread, prefer the sent one (it's the more recent signal) |
| Entity extraction misidentifies (e.g. "Q3" tagged as a project) | Mark entities as `kind: "unknown"` if Claude isn't confident; UI only renders chips for known kinds |
| `reply_outcome='rejected'` data is sparse early on | Default the pill to "Completed" until enough Rejected signals accumulate to be meaningful |

## 10. One-paragraph TL;DR for Cursor

> Reverse-engineering Nummo's Gmail surfacing reveals five patterns: real-time polling so emails surface within a minute, specific imperative titles that always include the topic, synthesized 1-2 sentence subtitles explaining who/what/why, extraction from the user's SENT folder for self-made commitments, and a three-state reply lifecycle (Approved / Rejected / Completed). taskbash already has the dedup + extraction pipeline; what's missing is (1) the polling cadence — add an Inngest gmail-poll function with incremental sync via Gmail historyId, (2) the title + subtitle prompt upgrade in lib/extract/gmail.ts plus a new `subtitle` column on items, (3) a commit-mode extractor pass on `in:sent newer_than:7d`, (4) a `reply_outcome` column with corresponding UI pills in ClearedTab, and (5) entity chips rendered in the subtitle. Six blocks of work, A-F; ship Block B first (biggest visual win, no infra), then A (the real-time behavior), then the rest in order.

## 11. Things to NOT do

- **Don't reinstate auto-complete-vanished.** It's intentionally disabled. Read CONTEXT.md section 14.
- **Don't change `semantic_hash` inputs.** It's the dedup primary key.
- **Don't use em-dashes.** Anywhere. UI or prompts. The codebase has style enforcement for this.
- **Don't roll your own UI primitives** when `app/_components/ui/` has shadcn versions.
- **Don't push directly to main.** Always: dev → preview → main.

Open `CONTEXT.md` alongside this doc for the architectural overview. Open the specific `lib/extract/gmail.ts` + `lib/digest/run.ts` files when implementing.
