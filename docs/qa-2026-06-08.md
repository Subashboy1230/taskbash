## Severity-ordered bug list

### P0 — Functional / data integrity

**B1. React hydration error #418 fires app-wide**
- Pages confirmed: `/today`, `/activity`, `/observability` (likely all routes)
- Symptom: Console throws `Minified React error #418` ("Hydration failed because the initial UI does not match what was rendered on the server") on every page load. Browser re-mounts the tree, causing brief flashes and lost interactions.
- Once Sentry is wired in production (Ticket 1 already shipped to dev), this error will pour into Sentry's free-tier quota at scale.
- Was flagged in Round 3 QA (task #166) as "fixed" — regression confirmed.
- Likely cause: locale/time-sensitive output (e.g., `new Date().toLocaleString()`) computed differently on server vs client.
- Fix candidates: gate locale-sensitive computed strings behind `useEffect` (client-only render), or `suppressHydrationWarning` if cosmetic.

**B2. "Morning digest ran" Inngest job stuck in Running status**
- Page: `/activity`
- Symptom: An "Agent run" row for "Morning digest ran" timestamped 11:12 PM shows status `Running` with `0 sources`, but child Prep tasks were created at 11:13 PM (one minute later) and marked `Completed`. The parent run never flipped to `Completed`.
- Either the run finished and the status update failed, OR the run actually died after spawning children. Hard to tell from the UI.
- Risk: if status-update is failing silently, you have no way to know which crons completed vs hung.
- Fix candidate: in the morning-digest Inngest function, ensure the final `step.run` updates the run row to `completed` and that this happens in a `try/finally` so a downstream error doesn't leave it stuck.

### P1 — Visible polish / inconsistency

**B3. HTML entities not decoded in task subtitles**
- Page: `/today` Open tab, P0 - Critical section
- Symptom: Task "Reply to Subash Rajaseelan re: 30 min with Subash (Kathryn Kelly)" shows subtitle text `Here&#39;s my calendar link` instead of `Here's my calendar link`. Raw HTML entity rendered as plain text.
- Cause: Gmail draft body is HTML-escaped (probably at storage), but rendered with raw text mode. Either double-escape or no decode at render.
- Fix: decode entities at render time. Quick helper: `text.replace(/&#39;/g, "'").replace(/&quot;/g, '"').replace(/&amp;/g, '&')`. Apply in the subtitle component, not at storage.

**B4. /settings/whatsapp is missing the entire app sidebar**
- Page: `/settings/whatsapp`
- Symptom: This page renders a floating centered form with NO left sidebar (no Home/Profile/Connections/Activity/Network nav), no "Back to today" link, no breadcrumb. Compare to `/settings/functions` which has full sidebar.
- Effect: user lands here, has no way to navigate except browser back. Confusing.
- Fix: wrap `app/(app)/settings/whatsapp/page.tsx` (or its layout) in the same `AppShell` / `AppSidebar` layout that wraps every other route. Likely a missing layout file inside the `whatsapp` directory, OR a route group that excludes the shared layout.

**B5. /profile page defaults to Stats tab instead of Overview**
- Page: `/profile`
- Symptom: Direct navigation to `/profile` opens with the "Stats" tab active, not "Overview" (which is the first tab and shows the user's name, source connections, and headline numbers). Clicking "Overview" works fine — content is there.
- Cause: likely persisted tab state in URL/query string or sessionStorage that wasn't reset on cold load. Or the default tab in the Tabs component is `defaultValue="stats"`.
- Fix: in the Tabs component for `/profile`, set `defaultValue="overview"` (and ignore any persisted state on cold mount).

**B6. Slop-rate-by-source chart is empty on /profile/stats**
- Page: `/profile`, Stats tab
- Symptom: The "Slop rate by source (last 30 days)" section renders the heading and a chart-shaped container but the container is empty. No bars, no axes, no "no data" placeholder. Just blank space.
- Either: chart library failed to render, OR there genuinely is no slop-by-source data, OR the data fetch errored silently (the "silent .catch swallows" pattern from polish ticket #5).
- Fix candidates: (1) wrap in a try/catch that renders a friendly "no slop data this month" empty state; (2) verify the chart component receives data — `console.log(data)` in the chart wrapper to confirm. If data is genuinely empty for this account, render a non-empty placeholder explaining what would appear.

**B7. Right calendar panel inconsistent across pages**
- Pages affected: `/activity` shows `No events scheduled today.` Other pages with the same right panel (`/handled`, `/observability`, `/connections`, `/profile`, `/network`, `/settings/functions`, `/today`) correctly show today's calendar events.
- Symptom: `/activity` uniquely fails to render today's events in the right panel — looks like the data fetch for the events list doesn't fire on that route specifically.
- Fix: trace which component fetches calendar events for the right panel. The component is probably rendered by a per-page layout. Confirm `/activity`'s layout includes the events fetcher, or that the event-fetcher hook runs on this route too.

**B8. Tab counts on /today appear to mutate inexplicably (Prep ↔ Cleared)**
- Page: `/today`
- Symptom: On first load, counts were `Open 126 · Prep 11 · Cleared 6 · Unread 17`. After interacting with a task row near the top of the list (which opened the detail panel for a Prep item), counts shifted to `Open 126 · Prep 6 · Cleared 11 · Unread 17`. Prep and Cleared appear to have swapped values.
- Could be a real reclassification triggered by the click (e.g., the click marked something cleared and something prep at once), OR a label/value swap bug in the tab-count fetch.
- Confidence: Medium. Need deliberate repro to confirm. Recommend instrumenting the tab-count endpoint to log which segment each count belongs to.
